03 Most Important Interview Questions

🔵 Angular Change Detection

BEGINNER
What is Angular's Change Detection and what triggers it?
✅ Expected: Mention zone.js, async events (clicks, XHR, timers), how Angular walks the component tree from root to leaf.
INTERMEDIATE
What is OnPush Change Detection? When would you use it and what are its gotchas?
✅ Expected: Input references must change (not mutate), async pipe triggers CD automatically, manual markForCheck() / detectChanges() when needed. Gotcha: shared mutable state silently breaks OnPush.
SENIOR
How do Angular Signals change the change detection model? What's the long-term direction for zone.js?
✅ Expected: Signals are fine-grained reactive — only components that read a signal re-render. Zone.js will become optional (zoneless mode). effect() replaces some RxJS patterns. computed() is memoized. The goal is signal-based components with no zone overhead.
TRICKY FOLLOW-UP
If you have OnPush on a component but it still updates on every tick, what's likely wrong?
✅ Expected: A parent component without OnPush is triggering CD that cascades down. Or a shared mutable object is being mutated (not replaced). Or an observable is emitting and async pipe is triggering markForCheck.

🟣 RxJS

BEGINNER
What's the difference between Subject, BehaviorSubject, and ReplaySubject?
✅ Expected: Subject — no initial value, late subscribers miss emissions. BehaviorSubject — holds latest value, new subscribers get it immediately. ReplaySubject — buffers N values for late subscribers.
INTERMEDIATE
switchMap vs mergeMap vs concatMap vs exhaustMap — when would you use each?
✅ Expected: switchMap cancels previous (search typeahead). mergeMap runs all concurrently (parallel requests). concatMap queues (ordered processing). exhaustMap ignores new while active (submit button). This is a very common senior question.
SENIOR
Walk me through how you prevent memory leaks in RxJS subscriptions at scale in your platform.
✅ Expected: takeUntil with a destroy$ Subject, async pipe in templates (auto-unsubscribes), take(1) for one-shot calls, Angular's DestroyRef with takeUntilDestroyed() in newer versions.

🟠 Micro Frontends

BEGINNER
Why did you choose Micro Frontend architecture for your platform?
✅ Expected: Independent team deployments, separate release cycles, different teams owning different app sections, avoid monolith bottlenecks in CI/CD.
INTERMEDIATE
How do you handle shared dependencies like Angular and RxJS across MFE remotes?
✅ Expected: The "shared" config in Module Federation — you define singletons (singleton: true, strictVersion: true/false). Angular must be a singleton. You decide per-library whether to enforce strict version matching or allow ranges.
SENIOR
How do you share authentication state between your host shell and remote MFE apps?
✅ Expected: Shared auth service via a shared library remote, or a custom event bus, or browser storage with token sharing. JWT stored in memory (not localStorage for XSS reasons) in shell, passed to remotes via a shared module.
TRICKY FOLLOW-UP
What happens when a remote MFE fails to load? How do you handle it gracefully?
✅ Expected: Dynamic import wrapping in try/catch, fallback component or error boundary component, manifest-based loading with retry logic. Discuss how you communicated this failure state to the user.

🔴 Performance Optimization

INTERMEDIATE
Walk me through how you achieved a 35% UI rendering improvement on 100k+ node tree data.
✅ Expected: This is YOUR story — structure it as: Problem → Root Cause Analysis → Solutions Applied → Result. Mention virtual scrolling (CDK), OnPush + trackBy, lazy-loading tree node children, on-demand rendering of collapsed branches. Be specific about how you measured the before/after.
SENIOR
How did you identify which components were causing the most change detection overhead?
✅ Expected: Angular DevTools profiler — shows component tree, number of change detections, time spent. Chrome Performance tab for paint/layout thrashing. Console.count inside ngOnChanges/ngDoCheck for debugging.

🟢 Leadership / Behavioral (Senior-Level Must Prep)

INTERMEDIATE
Tell me about a time you had to balance technical debt with feature delivery pressure.
✅ Expected: Concrete example, not generic. Use your Angular upgrade project — you were modernizing deprecated APIs while also delivering new features. How you made the tradeoff transparent to stakeholders.
SENIOR
How did you manage the Angular v6 to v20 upgrade without production regressions?
✅ Expected: Phased migration strategy, feature-flag isolation, incremental module updates, Cypress E2E as regression safety net, tested on staging env for each major version jump, rollback plan per phase.

Coding Interview Communication Pattern

  1. Restate the problem — "So if I understand correctly, I need to..."
  2. Clarify edge cases — "What should I return if the input is empty? Can there be duplicates?"
  3. State your approach first — "I'm thinking brute force is O(n²). I can optimize to O(n) using a hashmap."
  4. Code while narrating — explain what each line does as you write
  5. Test with example — trace through your solution with their example
  6. Mention complexity — "This is O(n) time, O(n) space for the hashmap"
06 Real-World Case Studies to Master
📊 Scenario: 100k Node Tree — Slow Rendering

Interviewer asks: "Your UI is freezing when users expand a tree with thousands of nodes. How do you debug and fix it?"


Your answer structure:

  • Diagnose: Chrome Performance tab to find long tasks. Angular DevTools to see CD cycle count. Identify which component triggers the most re-renders.
  • Root causes: All nodes rendered in DOM at once. Default Change Detection running on every user event across the whole tree.
  • Fixes applied: (1) CDK Virtual Scroll — only render visible rows. (2) OnPush on all tree node components. (3) trackBy to prevent DOM re-creation on data refresh. (4) Lazy load child nodes on expand event — don't load entire tree upfront.
  • Result: Reduced initial render time, browser stays responsive on expand/collapse.
🏗️ Scenario: Micro Frontend — Remote App Fails to Load

Interviewer asks: "In your MFE platform, one remote app throws an error during dynamic import. What happens and how do you handle it?"


  • Wrap dynamic imports in try/catch — render an error fallback component instead of crashing the shell
  • Use Angular's ENVIRONMENT_INITIALIZER or APP_INITIALIZER to pre-validate remote manifests on app startup
  • Show a user-friendly "this section is temporarily unavailable" message
  • Log the error to your monitoring system (Sentry/Datadog) with the remote name and version info
  • Design the shell so other MFE apps continue working — single remote failure shouldn't bring down the platform
⚡ Scenario: Build Time Optimization

Interviewer asks: "How did you reduce Angular build time by 69%?"


  • Problem: Single monolithic Angular application, all feature modules eagerly loaded, large shared module re-compiled on every change
  • Analysis: Webpack Bundle Analyzer to find the largest chunks. Identified circular dependencies. Found large shared modules that had no lazy-loading strategy.
  • Actions: (1) Moved to Module Federation — remotes compile independently. (2) Extracted truly shared code to a shared library remote. (3) Eliminated circular deps. (4) Enabled persistent build cache.
  • Result: Each app compiles only its own code. HMR only rebuilds the changed remote, not the whole platform.
🐛 Scenario: Production Debugging — Data Not Updating

Interviewer asks: "Users report that a list stops updating after the first few clicks. You can't reproduce in dev. What's your approach?"


  • Check if the issue is reproduction-environment-specific (data-driven vs code-driven)
  • Add temporary console logging to the subscription chain to check if emissions stop
  • Check for an exhaustMap or first() operator that stops listening after one emission
  • Check if the component was destroyed and subscription left dangling
  • Look at error boundary / global error handler — maybe an error is silently swallowing subsequent emissions
  • Check zone.js — is something running outside Angular's zone and updates not being detected?
🔐 Scenario: Security Vulnerability in Release Pipeline

Interviewer asks: "You mentioned resolving Xray security vulnerabilities. Walk me through that process."


  • JFrog Xray scans dependencies in the artifact repository and flags CVEs
  • Triage vulnerabilities by severity (critical/high/medium) — critical must be resolved before release
  • Check if the vulnerable dependency is directly used or transitive (nested dep)
  • Fix: update the direct dependency version, or use npm/yarn resolutions to force a safe version of a transitive dep
  • Rebuild, re-scan to confirm CVE is resolved, document the fix for audit trail
  • Added Xray scan as a mandatory CI gate — builds with critical CVEs now fail automatically
🗄️ Scenario: PostgreSQL JSONB for PDM Data

Interviewer asks: "Why did your team use JSONB instead of normalized tables for some PDM data?"


  • PDM (Product Data Management) entities have variable attribute schemas — different equipment types have different field sets
  • JSONB lets you store schema-flexible attributes without adding columns for every possible field
  • GIN index on the JSONB column allows fast lookups on nested fields
  • Tradeoff: JSONB is harder to query, no foreign key constraints inside JSON, harder to migrate if schema evolves predictably
  • Best fit when: schema is genuinely variable, read patterns are known, you can index the commonly queried paths
07 Mock Interview Training

How Senior Engineers Answer — The STAR+ Pattern

For every technical story, structure as: Situation → Problem/Root Cause → Actions (specific, your contribution) → Result (metric) → Learning


The key difference between a mid-level and senior answer is the depth on "Actions" — seniors explain WHY they chose a specific approach over alternatives, and what tradeoffs they consciously made.

How to Answer When You Only Partially Implemented Something

Honest framing script

"My direct involvement was on the frontend side — I owned [X specific thing]. The backend piece was handled by the Java team; I worked closely with them on [API contracts / data structure / integration points]. I can speak to how it worked from the integration side, though I didn't write the backend service code myself."

This answer is far more credible than pretending you owned the whole stack. Interviewers can tell when someone is bluffing on backend depth. Owning your frontend expertise confidently while being honest about collaboration is a senior-level signal, not a weakness.

How to Explain Architecture Simply

The "Explain to a New Team Member" test

Before any interview, practice explaining your MFE architecture as if onboarding a new engineer. If you can explain it in 3 sentences without jargon, you truly own it. Example:


"We have a shell application that acts as the container. It dynamically loads 4 independent Angular apps at runtime using Webpack Module Federation — each team owns and deploys their own app independently. The shell handles the routing, auth, and shared utilities like design system components."
