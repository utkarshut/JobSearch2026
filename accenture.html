Collections & Data Structures
What is the difference between HashMap and ConcurrentHashMap?
High freq
Answer
HashMap is not thread-safe. Multiple threads accessing it simultaneously can cause data corruption or infinite loops. ConcurrentHashMap is thread-safe — it uses segment-level locking (Java 7) or CAS operations (Java 8+) instead of locking the whole map, giving much better concurrent performance.

Key differences:
  • HashMap allows one null key and multiple null values; ConcurrentHashMap does not allow null key or value
  • ConcurrentHashMap is best for multi-threaded environments; HashMap for single-threaded
  • ConcurrentHashMap has higher memory overhead due to concurrency structure

Your angle: "In my enterprise Angular + Java platform, backend microservices handling concurrent API requests from multiple MFE modules used ConcurrentHashMap for in-memory caching of session-specific config — HashMap would have caused race conditions under simultaneous user load."

ArrayList vs LinkedList — when do you use which?
Likely
Answer
ArrayList — backed by a dynamic array. Fast random access O(1), slow insertion/deletion in the middle O(n). Use when you read more than you write.

LinkedList — doubly linked list. Slow random access O(n), fast insertion/deletion at both ends O(1). Use when you frequently add/remove from the beginning or middle.

For application support scenarios: when processing large log queues (FIFO), LinkedList is better; when indexing parsed data rows, ArrayList wins.
What happens when you put a duplicate key in a HashMap?
Rapid-fire
Answer
The new value overwrites the old value for that key. put() returns the previous value (or null if no previous value existed). The key count stays the same — no duplicate keys in HashMap.

Follow-up they may ask: "How does HashMap handle hash collisions?" → Java uses chaining (linked list in each bucket, converted to a balanced tree — red-black tree — when bucket size exceeds 8 in Java 8+) to handle collisions.

Exception Handling
Difference between checked and unchecked exceptions. Give examples.
Always asked
Answer
Checked exceptions — must be handled at compile time (try-catch or throws). They represent external conditions the program should recover from. Examples: IOException, SQLException, FileNotFoundException.

Unchecked exceptions (RuntimeExceptions) — not required to be declared or caught. They represent programming bugs. Examples: NullPointerException, ArrayIndexOutOfBoundsException, ClassCastException.

Error — severe JVM-level issues not meant to be caught: OutOfMemoryError, StackOverflowError.

Your angle: "In production debugging at Halliburton, I traced critical incidents caused by uncaught NullPointerExceptions in async data pipelines — we added defensive null-checks and input validation at the REST API layer which reduced P1 incidents significantly."

What is the difference between final, finally, and finalize?
Classic rapid-fire
Answer
final — keyword. Variable = value can't change. Method = can't be overridden. Class = can't be subclassed.

finally — block in try-catch-finally. Always executes whether exception occurs or not. Used to release resources (close DB connections, files).

finalize() — method in Object class, called by the garbage collector before an object is destroyed. Deprecated in Java 9, removed in Java 18. Use try-with-resources instead for cleanup.
String & JVM
Why is String immutable in Java? What is the String Pool?
Very likely
Answer
Immutability: String objects cannot be changed after creation. Any modification creates a new String object. Reasons: security (used in class loading, network connections), thread safety (immutable objects are inherently thread-safe), caching (hashCode can be cached).

String Pool: A special area in the heap (PermGen in older versions, now in main heap via Metaspace) where Java stores string literals. When you write String s = "hello", Java checks the pool first — if "hello" exists, it reuses the reference. new String("hello") always creates a new object outside the pool.
String a = "hello"; // Pool String b = "hello"; // Same reference as a String c = new String("hello"); // New object System.out.println(a == b); // true System.out.println(a == c); // false System.out.println(a.equals(c)); // true

Streams & Lambdas
What is the Stream API? How is it different from Collections?
Always asked
Answer
Collection stores data in memory and supports CRUD. You interact with the data directly.
Stream processes data through a pipeline of operations without modifying the source. It is lazy — intermediate operations (filter, map) are not executed until a terminal operation (collect, forEach, count) is called.

Key stream features: lazy evaluation, can be parallelized with parallelStream(), can only be consumed once.
// Filter even numbers and collect to list List<Integer> evens = numbers.stream() .filter(n -> n % 2 == 0) .collect(Collectors.toList()); // Map and reduce int sum = numbers.stream() .mapToInt(Integer::intValue).sum();

Your angle: "Very similar to RxJS observables in Angular — both are lazy, pipeline-based, and support operators like filter/map. I used Stream API to process large JSON payloads from REST APIs in the Java backend layer of our Halliburton platform."

What is a Functional Interface? Name some built-in ones in Java 8.
High freq
Answer
A functional interface has exactly one abstract method. Annotated with @FunctionalInterface. They enable lambda expressions.

Built-in ones in java.util.function:
  • Predicate<T> — takes T, returns boolean. Used in stream filter()
  • Function<T,R> — takes T, returns R. Used in map()
  • Consumer<T> — takes T, returns nothing. Used in forEach()
  • Supplier<T> — takes nothing, returns T
  • BiFunction<T,U,R> — takes two inputs, returns result
What is Optional? Why was it introduced?
Likely
Answer
Optional<T> is a container object that may or may not contain a value. Introduced to reduce NullPointerException risks and make null handling explicit.

Optional<String> name = Optional.ofNullable(user.getName()); name.ifPresent(n -> System.out.println("Name: " + n)); String result = name.orElse("Unknown"); String val = name.orElseThrow(() -> new RuntimeException("No name"));

Support angle: "In production, a major class of NullPointerExceptions in our REST API handlers was eliminated after we refactored return types to use Optional — this came up directly in a root-cause analysis I led."

What is the difference between map() and flatMap() in Streams?
Senior-level check
Answer
map() — transforms each element to one output. Returns Stream<Stream<T>> if the function returns a stream.
flatMap() — transforms and then flattens. If each element maps to a stream, flatMap merges them into one single stream.

// map: List of lists stays nested List<List<Integer>> nested = ...; nested.stream().map(List::stream); // Stream<Stream<Integer>> // flatMap: flattens to one stream nested.stream().flatMap(List::stream); // Stream<Integer>

Abstract class vs Interface — when do you use which?
Always asked
Answer
Abstract class: Use when classes share common state/behaviour and there's a clear IS-A relationship. Can have constructor, instance variables, concrete + abstract methods. A class can extend only ONE abstract class.

Interface: Use for defining a contract/capability, especially for unrelated classes. In Java 8+ can have default and static methods. A class can implement MULTIPLE interfaces — enables multiple inheritance of type.

Rule of thumb: If you need to share code → abstract class. If you need a contract without shared state → interface.

Your angle: "In the Micro Frontend shell, Java backend services followed interface-driven design — each domain's service contract was an interface, enabling mocking in tests and swapping implementations with zero frontend impact."

What is method overloading vs method overriding?
Rapid-fire
Answer
Overloading — same method name, different parameters, in the same class. Resolved at compile time (static polymorphism).
Overriding — child class provides its own implementation of a method defined in the parent class. Same signature. Resolved at runtime (dynamic polymorphism). Use @Override annotation.

Pattern: Multithreading is heavily asked for senior Java candidates at Accenture. Focus on concepts + real production scenarios. They especially like questions about deadlock, thread pool, and synchronized.

What is a deadlock? How do you prevent it?
Very likely
Answer
Deadlock — Thread A holds Lock 1, waits for Lock 2. Thread B holds Lock 2, waits for Lock 1. Neither can proceed — circular wait.

Prevention strategies:
  • Always acquire locks in a consistent order across all threads
  • Use tryLock() with timeout instead of indefinite waiting
  • Minimise the scope of synchronized blocks
  • Use higher-level concurrency utilities (java.util.concurrent) instead of low-level synchronized

Production angle: "At Halliburton, an async processing pipeline for document workflows had intermittent hangs in staging. Root cause was a deadlock between two service threads competing for a shared cache lock and a DB connection lock. We resolved it by enforcing consistent lock acquisition order and switching to ConcurrentHashMap."

What is the difference between synchronized method and synchronized block?
Common
Answer
synchronized method — locks the entire object (or class for static). Only one thread can execute ANY synchronized method on that object at a time. Coarse-grained.

synchronized block — locks only the specified object/resource. More fine-grained, better performance because other parts of the method can run concurrently.
// Synchronized block — better performance public void process() { // non-critical code runs freely synchronized(this.cache) { // only this section is locked cache.update(); } // more non-critical code }
What is a thread pool? Why use ExecutorService instead of creating threads manually?
High freq for senior
Answer
Thread pool — a pool of pre-created threads ready to execute tasks. Avoids the overhead of creating/destroying threads for every task.

ExecutorService is Java's thread pool framework. Benefits:
  • Thread reuse — no repeated creation overhead
  • Controlled concurrency — limit max threads to prevent resource exhaustion
  • Task queue management — tasks wait if all threads are busy
  • Graceful shutdown via shutdown() and awaitTermination()
ExecutorService pool = Executors.newFixedThreadPool(10); pool.submit(() -> processDocument(doc)); pool.shutdown();

Your angle: "In our async processing backend for document workflows, we used a fixed thread pool via ExecutorService to handle parallel document processing — without it, unrestricted thread creation would have exhausted server memory under peak load."

Query writing — expect at least one live query
Write a query to find the second highest salary from an Employee table.
Classic — always asked
Answer — know all 3 approaches
-- Approach 1: Subquery (most common answer) SELECT MAX(salary) FROM Employee WHERE salary < (SELECT MAX(salary) FROM Employee); -- Approach 2: LIMIT/OFFSET (PostgreSQL, MySQL) SELECT DISTINCT salary FROM Employee ORDER BY salary DESC LIMIT 1 OFFSET 1; -- Approach 3: DENSE_RANK window function (best approach) SELECT salary FROM ( SELECT salary, DENSE_RANK() OVER (ORDER BY salary DESC) AS rnk FROM Employee ) ranked WHERE rnk = 2;

Show off: Mention DENSE_RANK as the preferred production approach — it handles ties correctly and scales. Your PostgreSQL experience with window functions is a real differentiator here.

What is the difference between INNER JOIN, LEFT JOIN, and FULL OUTER JOIN?
Always tested
Answer
INNER JOIN — returns only rows where there is a match in both tables.
LEFT JOIN — returns all rows from the left table + matching rows from the right. Non-matching right rows → NULL.
RIGHT JOIN — opposite of LEFT JOIN.
FULL OUTER JOIN — returns all rows from both tables. Non-matching rows on either side → NULL.

Application support use case: "When diagnosing a data discrepancy, I used LEFT JOIN to find records in the workflow table that had no corresponding entry in the document table — orphaned records from a failed migration."
What are database indexes? When would you and wouldn't you use them?
Senior-level
Answer
Index — a data structure (usually B-tree) that speeds up SELECT queries on a column at the cost of slower writes and extra storage.

Use indexes when: Columns used frequently in WHERE, JOIN, ORDER BY clauses; columns with high cardinality (many distinct values); large tables where full scans are slow.

Avoid indexes when: Small tables (full scan faster); columns with very low cardinality (e.g. boolean); heavily write-intensive tables (indexes slow down INSERT/UPDATE/DELETE).

Your PostgreSQL angle: "In our Halliburton PDM platform, performance profiling on the 100k+ node hierarchy queries revealed missing composite indexes on frequently joined foreign keys. Adding targeted indexes reduced query response time by over 60%. We also used EXPLAIN ANALYZE in PostgreSQL to identify slow query plans."

What is a stored procedure? When would you use one in application support?
Support-specific
Answer
A stored procedure is a precompiled set of SQL statements stored in the database and executed as a unit. Benefits: reduced network round trips, reusability, security (grant execute without exposing table structure).

Application support use: Running complex data fix scripts, batch cleanup of orphaned records, automating periodic data reconciliation jobs — without needing application code changes.

A Java application is running in production and suddenly becomes very slow. How do you diagnose it?
Near-certain
Your structured answer (5-step framework)
Step 1 — Confirm and scope: Is it all users or specific users/regions? Is it all operations or specific endpoints? Check monitoring dashboards — CPU, memory, GC activity, DB connection pool.

Step 2 — Check recent changes: Any deployment in the last 24 hrs? Config change? Data volume spike?

Step 3 — Look at logs: Application logs for exceptions, slow query logs in DB, GC logs for full GC pauses.

Step 4 — Profile the JVM: Take a thread dump (jstack) to see if threads are blocked/deadlocked. Take a heap dump (jmap) if memory is suspect. Use jstat for GC stats.

Step 5 — DB investigation: Check for slow queries, missing indexes, locking/blocking queries using pg_stat_activity (PostgreSQL) or equivalent.

Tie to your work: "Exactly this process helped me diagnose a 35%+ UI rendering slowdown at Halliburton — traced it to N+1 query patterns in the tree node API plus missing index on a frequently joined FK column."

A critical API is returning 500 errors intermittently. Walk me through your investigation.
Very likely
Answer
1. Reproduce reliably: Identify the pattern — specific payload? Time of day? Specific users? Use request IDs/correlation IDs to trace in logs.

2. Check server-side logs: Full stack trace of the 500. Is it a NullPointerException? DB timeout? External service failure?

3. Check downstream dependencies: Is the DB up and responsive? Is any third-party service timing out? Check connection pool exhaustion.

4. Check for data-specific issues: Does it fail for specific input? Edge case in data handling (null field, unexpected format).

5. Hotfix path: Can we add a null-check or retry logic quickly without a full release? Use feature flags to disable the broken feature while fix is prepared.
How do you approach identifying a memory leak in a Java application?
Senior-level
Answer
Signs: Application slows down over time, OutOfMemoryError in logs, heap usage keeps growing even after GC.

Investigation steps:
  • Monitor heap usage over time with JVM metrics or APM tool
  • Take heap dump: jmap -dump:format=b,file=heap.hprof <pid>
  • Analyse with Eclipse MAT or VisualVM — find which objects are growing unexpectedly
  • Common culprits: static collections holding references, unclosed DB connections, listeners not removed, caches without eviction policy

Your angle: "In our Redis caching layer integration, we identified a memory leak caused by cache entries with no TTL set — objects were never evicted. Adding TTL policy and fixing the Java-side cache manager resolved it."
